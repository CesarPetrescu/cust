## Findings

- 2026-07-12: Local `cc -std=c11 -Wall -Wextra -Werror` accepts `_Atomic(int *)`, `_Atomic(const int *)`, and `_Atomic(struct T *)`; the atomic-qualified object is the pointer slot, while the pointee type and its `const` qualification remain part of the pointer type. Cust previously stopped nested `_Atomic(type-name)` parsing before `*`. Implementation decision: parse one supported pointer star inside `_Atomic(...)`, map nested leading `const` to `points_to_const`, retain post-star qualifier metadata, and reject second-star, pointer-array, and function suffixes at their source tokens. Compiler-oracle fixture `atomic_pointer_type_specifiers.c` checks pointer use plus ABI-independent `sizeof`/`_Alignof` relationships. The same probe showed `_Atomic(ArrayAlias)` is rejected by native C but still accepted by Cust, so that narrower constraint is the next TODO.

- 2026-07-12: No external documentation was needed for void array/function type-name diagnostics. Root cause: dedicated `void` branches in `parse_cast()` and `parse_sizeof_like_type_name()` checked void pointers but otherwise immediately expected `)` or rejected bare `void`, so following function/array suffixes were ignored or reached generic delimiters. Implementation decision: preserve void-pointer detection first, then route `(` through the shared function-type suffix diagnostic and reject `[` with route-specific void-array wording before handling bare void. See `references/cust-void-array-function-type-name-diagnostics.md`.

- 2026-07-12: No external documentation was needed for pointer-to-array cast/type-query diagnostics. Root cause: direct scalar and aggregate array suffix paths parsed `[N]` and then checked only a second `[` or function `(` suffix before generic closing-delimiter handling, while `DeclType::Array` from typedef aliases already rejected a following `*` as pointer-to-array syntax. Implementation decision: reject `*` immediately after every parsed direct array suffix in casts and shared `sizeof`/`_Alignof` type-name parsing, reporting `pointer-to-array casts/<operator> types are not supported` at the star and preserving multidimensional-array, function-array, pointer-array, and valid one-dimensional routes.

- 2026-07-12: No external documentation was needed for multidimensional-array designator diagnostics. Root cause: both bounded and inferred/unbounded array designator parsers consumed one `[index]` and returned immediately, so every caller expected `=` next and a second `[` fell through to a generic missing-assignment diagnostic. Implementation decision: reject a following `[` in one shared post-index helper used by both designator parsers, reporting `multidimensional array designators are not supported` at the suffix while leaving one-dimensional designator evaluation, range rejection, and bounds checks unchanged. A follow-up probe found array-then-star cast/type-query forms still use generic closing-delimiter diagnostics and recorded them as the next candidate in `status/todo.md`.

- 2026-07-12: No external documentation was needed for multidimensional-array cast diagnostics. Root cause: every direct array compound-literal route in `parse_cast()` consumed one `[N]` suffix and then immediately expected `)` (or checked a function suffix), unlike `parse_sizeof_like_type_name()`, which already rejects a second `[` explicitly. As a result, direct scalar, named/anonymous aggregate, and array-typedef forms fell through to a generic closing-parenthesis error. Implementation decision: apply one shared metadata-only suffix guard after the first cast array suffix and report `multidimensional array casts are not supported` at the second `[` while preserving supported one-dimensional array compound literals and adjacent function-array diagnostics.

- 2026-07-12: No external documentation was needed for pointer-function type-name diagnostics. Root cause: after parsing a supported one-level pointer suffix, `parse_cast()` and `parse_sizeof_like_type_name()` expected a closing parenthesis or returned `SizeOfType::Pointer` without checking a following abstract function declarator. Thus `(int *(void))0`, `sizeof(int *(void))`, and `_Alignof(IntPtr(void))` reached generic delimiter errors. Implementation decision: reuse `reject_function_type_suffix(...)` after every supported pointer route (direct scalar/named/anonymous aggregate and pointer typedef) while retaining its existing `(*` exception for dedicated parenthesized-pointer diagnostics. See `references/cust-pointer-function-type-name-diagnostics.md`.

- 2026-07-12: No external documentation was needed for function-array type-name diagnostics. Root cause: `parse_cast()` and `parse_sizeof_like_type_name()` guarded a function declarator immediately after the base type, but ordinary scalar and aggregate array suffix parsing consumed `[N]` and delegated the following `(` to a generic closing-parenthesis helper. Implementation decision: reuse `reject_function_type_suffix(...)` immediately after parsed array suffixes in casts, `sizeof`, and `_Alignof`; this retains the established function-type wording/location and does not alter valid array compound literals.

- 2026-07-11: No external documentation was needed for pointer-array `sizeof`/`_Alignof` diagnostics. Root cause: `parse_sizeof_like_type_name()` checked an array suffix after pointer typedef aliases but returned `SizeOfType::Pointer` immediately after direct scalar/named/anonymous aggregate `*`, leaving `[` for the generic closing-parenthesis helper. Implementation decision: add the same post-star `LBracket` guard to all three direct type-query branches, reporting `pointer array <operator> types are not supported` at `[` and preserving ordinary array type operands, post-star qualifiers, and the pointer-to-pointer boundary.

- 2026-07-11: Local `cc -std=c11 -Wall -Wextra -Werror -fsyntax-only` rejects `sizeof(restrict int)`, `_Alignof(restrict int)`, and `(restrict int)0` as invalid uses of `restrict`. Root cause: declaration parsing called `reject_leading_restrict_qualifier()`, but `parse_sizeof_like_type_name()` and `parse_cast()` consumed leading qualifiers directly, silently accepting `restrict` as no-op scalar metadata. Implementation decision: invoke the existing guard before those two direct type-name qualifier-consumption paths, preserving valid post-star pointer `restrict` syntax and the existing declaration/parameter/field diagnostic text.

- 2026-07-11: No external documentation was needed for unsupported array range designator diagnostics. Root cause: `...` lexes as three `Token::Dot` values; after a valid constant index, each shared array-designator parser called the generic closing-bracket helper, producing `expected ']' after array designator, found Dot`. Implementation decision: reuse the existing conservative three-dot lookahead (`starts_ellipsis`) in a post-index guard shared by bounded and unbounded array-designator parsing, reporting `array range designators are not supported` at the first dot. Exact-output interpreter coverage includes declared arrays, unsized array compound literals, and aggregate scalar-array field path designators; no GNU range-designator runtime support was added.

- 2026-07-11: No external documentation was needed for aggregate-initializer array-designator-start diagnostics. Root cause: `parse_struct_initializer()` permits dot-led field designators but, unlike array initializer paths, had no guard for a bare `Token::LBracket`, so `(struct Point){[}` reached scalar-expression parsing and emitted generic `expected expression, found LBracket`. Implementation decision: reject that token at the aggregate initializer element boundary with the existing contextual initializer wording, while preserving `[` designators in scalar/aggregate array initializer parsers and `.` field designators in aggregate initializers.

- 2026-07-11: No external documentation was needed for `sizeof` invalid-start diagnostics. Root cause: `reject_missing_sizeof_operand()` handled delimiter-like and selector/brace/bracket/question invalid starts together, so `sizeof([)`, `sizeof(?)`, `sizeof(:)`, `sizeof(.)`, `sizeof(->field)`, and `sizeof({1})` produced generic `found ...` diagnostics or fell through to primary-expression parsing. Implementation decision: keep delimiter-only missing operands (`sizeof()`, `sizeof(,)`, `sizeof;`) as `found ...`, but route bracket/question/colon/selector/brace starts to `expected sizeof operand before '<token>'`, matching adjacent `_Alignof`, `_Alignas`, `_Atomic(type-name)`, and integer-constant boundary style.

- 2026-07-11: No external documentation was needed for `_Atomic(type-name)` selector/colon invalid-start diagnostics. Root cause: `parse_decl_type_with_embedded_qualifiers()` handled `[`/`?`/`{`/keyword invalid starts contextually, then left `:`, `.`, and `->` in the generic delimiter/selector fallback. Implementation decision: reuse `integer_constant_invalid_start_or_selector_label()` for `_Atomic(type-name)` when the next token is not a valid type-name start, matching the existing parser-trust `before '<token>'` style used by `_Alignof`, `_Alignas`, and integer-constant boundaries while preserving valid bare `_Atomic` and `_Atomic(type-name)` parsing.
- 2026-07-11: No external documentation was needed for `_Alignas` / `_Static_assert` colon invalid-start diagnostics. Root cause: both operand guards handled delimiters, braces, brackets, selectors, and keyword invalid starts, but still delegated `Token::Colon` to generic expression parsing. Implementation decision: route both guards through `integer_constant_invalid_start_or_selector_label()` so colon shares the parser-trust `before ':'` diagnostic style already used in integer-constant boundaries, while leaving valid `_Alignas(type-name/expression)` and `_Static_assert(condition, message)` routes unchanged.

- 2026-07-10: No external documentation was needed for colon-token parameter and integer-constant parser-trust diagnostics. Root cause: the function-parameter guard covered commas, brackets, question, and braces but not semicolon/colon/selector starts, and shared integer-constant invalid-start labelling covered brace/selector starts but not colon starts. Implementation decision: add semicolon/colon/dot/arrow to initial and after-comma parameter-list guards while checking `...` before Dot/Arrow guards so variadic-parameter diagnostics remain targeted; include `Token::Colon` in `integer_constant_invalid_start_or_selector_label()` so fixed array lengths, array designator indexes, enum values, and switch case labels report contextual `before ':'` diagnostics.

- 2026-07-10: No external documentation was needed for subscript comma expressions. Root cause: `parse_index_expr()` parsed only `parse_logical_or()` before requiring `]`, so direct C subscript expressions such as `values[i++, 2]` failed with `expected ']' after array index, found Comma` even though Cust already supports comma expressions elsewhere and parenthesized comma expressions in indexes. Implementation decision: keep the existing first-operand/index-start path to preserve diagnostics like `values[0 = 3;`, then parse comma tails into `Expr::Comma` with normal comma-operator missing-RHS diagnostics. See `references/cust-subscript-comma-expressions.md`.

- 2026-07-10: No external documentation was needed for colon-token parser-trust diagnostics. Root cause: several route-local expression-boundary guards intentionally handled delimiters, brackets, selectors, braces, and keywords but omitted `Token::Colon`, so malformed forms such as `callee(:)`, `values[:]`, `int values[2] = {:}`, `return :;`, `if (:)`, `switch (:)`, and `return (:);` delegated to generic primary-expression parsing. Implementation decision: add narrow colon guards only at function-call, array-index, braced-initializer, return/control/switch, for-clause, and grouped-expression boundaries while preserving valid conditional-operator colon parsing and statement-label/case/default routing.

- 2026-07-10: No external documentation was needed for top-level invalid-start parser diagnostics. Root cause: `parse_program()` preserved exact unmatched delimiter and stray control-flow diagnostics, but otherwise returned the broad `unexpected token at top level: ...` fallback for malformed declaration-start tokens. Implementation decision: add a narrow file-scope guard for `Token::LBrace`, `Comma`, `Question`, `Dot`, and `Arrow` so malformed starts report `expected top-level declaration, found ...` without changing normal declaration/function routing.

- 2026-07-10: No external documentation was needed for brace-token `_Atomic`/parameter parser-trust diagnostics. Root cause: `_Atomic(type-name)` guarded `[`, `?`, and keyword/selector/delimiter starts but not `{`, and the initial function-parameter guard likewise omitted `{` even though after-comma parameter parsing already handled it. Implementation decision: add narrow `Token::LBrace` guards in `parse_decl_type_with_embedded_qualifiers()` for `_Atomic({)` and in `parse_params()` for leading `{` so `_Atomic({)`, `sizeof(_Atomic({))`, and `int add({ int b)` report contextual diagnostics without changing valid blocks or braced initializers.

- 2026-07-10: No external documentation was needed for brace-token parser-trust diagnostics. Root cause: `{` is valid as a statement-leading block and in supported initializer/compound-literal grammars, but malformed expression/type/integer-constant operand positions should not delegate `Token::LBrace` into generic primary/type parsing. Implementation decision: add narrow route-local `LBrace` guards for `_Alignas`, `_Static_assert`, `sizeof`, `_Alignof`, grouped expressions, array/index/integer-constant boundaries, control clauses, and operator operands so examples like `sizeof({1})`, `_Alignof({1})`, `values[{1}]`, `case {1}:`, and `return 1 ? {2} : 3` report contextual diagnostics while statement blocks and valid braced initializers remain unchanged.
- 2026-07-10: No external documentation was needed for selector-token integer-constant diagnostics. Root cause: parenthesized integer constants and ordinary expression routes already recognized malformed Dot/Arrow starts contextually, but array declarator lengths, array designator indexes, enum constant values, and switch case labels still delegated those selector tokens into generic integer-constant parsing. Implementation decision: add `integer_constant_invalid_start_or_selector_label()` and use it only in those integer-constant boundary guards so `int values[.field];`, `{[->field] = 1}`, `enum Bad { A = .field };`, and `case ->field:` report contextual `expected ... before '.'/'->'` diagnostics without changing expression-context selector-token wording.
- 2026-07-10: No external documentation was needed for selector-token array/string index diagnostics. Root cause: parser-trust guards covered Dot/Arrow starts in statement, return/operator/control/type-query/cast/function-call/initializer/grouped/unary routes, but `parse_index_expr()` still delegated malformed selector tokens inside array and string indexes to generic primary-expression parsing. Implementation decision: add narrow `Token::Dot` / `Token::Arrow` guards to `parse_index_expr()` so `values[.field]` and `"hi"[->field]` report contextual `expected array index expression...` diagnostics while preserving valid postfix selectors after real indexed expressions.

- 2026-07-09: No external documentation was needed for selector-token grouped-expression and unary-operand diagnostics. Root cause: parser-trust guards covered Dot/Arrow starts in statement, return/operator/control/type-query/cast/function-call/initializer routes, but grouped expression operands and prefix-unary operands still delegated malformed selector tokens to generic primary-expression parsing. Implementation decision: add narrow `Token::Dot` / `Token::Arrow` guards to the grouped-expression start check and `reject_missing_unary_operand(...)` while preserving valid postfix selector expressions after a real operand.
- 2026-07-09: No external documentation was needed for selector-token function-call/initializer diagnostics. Root cause: parser-trust guards covered Dot/Arrow starts in return/operator/control/type-query/cast routes, but function-call argument lists and braced initializer list starts still delegated malformed selector tokens to generic primary-expression parsing. The fix is route-local: treat Dot/Arrow as invalid call-argument starts and reject Arrow plus non-designator Dot in initializer element starts while preserving valid struct `.field` designators.
- 2026-07-09: No external documentation was needed for selector-token control/conditional expression diagnostics. Root cause: earlier parser-trust guards covered Dot/Arrow selector tokens in expression statements, return/operator RHS, `sizeof`, casts, `_Alignas`, and `_Static_assert`, but not in for-initializer starts, control-flow conditions, for increments, or conditional-operator branch operands. Malformed starts such as `for (.field; ; )`, `if (.field)`, `while (->field)`, `for (...; .field; ...)`, `for (...; ...; ->field)`, `return 1 ? .field : 2`, and `return 1 ? 2 : ->field` therefore fell through to generic primary-expression or for-increment fallbacks. Implementation decision: add narrow Dot/Arrow guards only at those expression-boundary start checks so valid postfix selector expressions after a real base expression remain unchanged.

- 2026-07-09: No external documentation was needed for qualifier-only declaration-type diagnostics. Root cause: declaration parsers consumed leading `const` / `volatile` / bare `_Atomic` qualifiers before asking the general type parser for a type, so malformed qualifier-only declarations such as `const global;`, `volatile local;`, `_Atomic value` parameters, aggregate fields, and typedef aliases lost the qualifier context and reported generic route-specific type errors. Implementation decision: have qualifier consumption optionally return the last consumed qualifier token, then reject non-type starts immediately in declaration/parameter/field/typedef contexts with `expected <context> after type qualifier '<qualifier>'...` while leaving valid qualified declarations and cast/sizeof/_Alignof qualifier diagnostics unchanged.

- 2026-07-09: No external documentation was needed for function-parameter keyword-start diagnostics. Root cause: `parse_params()` guarded delimiter, bracket, and `?` starts at the beginning of a parameter list and after separator commas, but delegated statement/control keywords such as `return` into `parse_decl_type(...)`, producing the generic `expected type, found Return`. Implementation decision: add narrow parameter-list guards for non-type keyword starts while preserving valid type-name starts, old-style identifier-list detection, variadic `...` diagnostics, and ordinary parameter parsing.

- 2026-07-09: No external documentation was needed for parenthesized integer-constant operand diagnostics. Root cause: `parse_integer_constant_primary()` delegated immediately to recursive integer-constant parsing after consuming `(`, so malformed `()`, `(,)`, `(:`, `(?)`, and `(return)` in enum initializers, array lengths, and switch labels produced generic `expected integer constant in parenthesized integer constant expression, found ...` messages. Implementation decision: add a narrow start-token guard for parenthesized integer-constant expressions while leaving valid parenthesized constants and post-left-operand comma-operator rejection unchanged.

- 2026-07-09: No external documentation was needed for `_Alignof` selector-token invalid-start diagnostics. Root cause: `reject_missing_alignof_type()` already guarded invalid `[` / `?` starts and keyword starts, but selector tokens `.` and `->` reached the generic non-type fallback and reported `expected _Alignof type, found Dot/Arrow`. Implementation decision: add narrow selector-token guards so malformed `_Alignof(.)` and `_Alignof(->field)` report contextual `expected _Alignof type before '.'/'->'` diagnostics while valid `_Alignof(type-name)` operands and integer-constant `_Alignof` folding remain unchanged.

Implementation decision: add narrow `_Atomic` type-argument guards before `parse_decl_type_with_embedded_qualifiers(...)` so declaration, parameter, `sizeof`, cast, and aggregate-field routes preserve contextual `_Atomic` diagnostics while keeping valid bare `_Atomic` qualifiers and `_Atomic(type-name)` specifiers unchanged.

Implementation decision: include `Token::Dot` and `Token::Arrow` in the route-local guards so malformed `_Alignas(.)`, `_Static_assert(->field, "msg")`, `sizeof(.)`, and `(int)->field` report contextual diagnostics while valid postfix selector use after aggregate/pointer expressions remains unchanged.

- 2026-07-09: No external documentation was needed for braced scalar initializer invalid-start diagnostics. Implementation decision: route scalar braced initializer operands through the narrower invalid-value guard and include selector tokens in that guard while keeping aggregate/array initializer designator starts unchanged.
- 2026-07-08: No external documentation was needed for integer-constant unary operand diagnostics. The root cause was parser routing: `parse_sizeof()` delegated empty/delimiter operands such as `sizeof()` and `sizeof;` to the generic expression parser, which reported `expected expression, found ...`; a narrow pre-parse delimiter guard now reports `expected sizeof operand, found ...` before expression parsing while `_Alignof` already had a targeted type-start diagnostic for `)`/`,` delimiters.
- 2026-07-03: No external documentation was needed for declaration initializer missing-expression diagnostics. The root cause was parser routing: declaration parsers consumed `=` and then delegated delimiter-leading inputs such as `int value = ;`, `int *slot = ,`, and `struct Point point = ;` to the generic expression parser, producing `expected expression, found ...`. A narrow delimiter guard now runs immediately after declaration initializer `=` on scalar, pointer, and aggregate expression-initializer paths, preserving omitted zero-initializers and braced aggregate initializers while reporting `expected initializer expression after '=' in ... declaration` at the actual delimiter.
- 2026-07-03: No external documentation was needed for excess braced initializer source locations. The root cause was parser-local: array, struct/union, scalar braced, and struct-array initializer parsers detected over-capacity entries before parsing the excess token but created unlocated `CustError::new(...)` diagnostics. These routes now capture `peek_located()` before returning the `too many initializers for ...` error so diagnostics point at the first excess initializer token while preserving existing missing-element and trailing-comma behavior.
- 2026-07-03: No external documentation was needed for excess compound-literal initializer source locations. The root cause matched ordinary braced initializers: fixed-size scalar array compound literal and aggregate-array compound literal parsers detected over-capacity positional entries before parsing the excess token but returned unlocated `CustError::new(...)` diagnostics. These routes now capture `peek_located()` and use `Parser::error_at(...)`, preserving existing fixed-size length enforcement while pointing at the first excess initializer token.
- 2026-07-03: No external documentation was needed for malformed path-designator diagnostics. The root cause was parser-local: path designator array indexes reuse the integer-constant-expression parser, so `.values[] = 1` reported the generic `expected array designator index, found RBracket`; both bounded and unbounded array designator helpers now special-case an immediate `]` and report `expected array designator index before ']'` at the bracket while keeping other delimiter-start diagnostics contextual.
- 2026-07-03: No external documentation was needed for fixed array and aggregate-array missing-initializer diagnostics. The root cause was parser routing: after consuming `=` for fixed scalar arrays and fixed aggregate arrays, the parser delegated immediately to the braced initializer parser, so delimiter-leading forms such as `int values[2] = ;` and `struct Point points[1] = ;` reported `expected '{' after ... initializer` rather than the declaration-level missing-initializer diagnostic used by scalar, pointer, and aggregate variable declarations. The array and struct-array initializer entry points now reuse the shared delimiter guard before accepting valid string/braced initializers.
- 2026-07-05: No external documentation was needed for delimiter-only array-length diagnostics. The root cause was parser-local: shared `expect_array_len()` had a special message for empty `[]` but delegated `[` followed by `)` to the generic integer-constant-expression parser, producing `expected array length, found RParen` across local declarations, parameters, aggregate fields, typedef aliases, and type-name queries. It now mirrors the empty-bracket guard for `)` and reports `expected array length before ')'` at the delimiter while preserving existing VLA-star and integer-constant-expression diagnostics.
- 2026-07-11: Native `cc -std=c11 -Wall -Wextra -Werror` accepts legal one-level post-star-qualified pointer type names (`sizeof(int * const)`, `_Alignof(int * volatile)`, `(int * const)0`) and also accepts nested pointer type spellings. Cust intentionally supports only one pointer level, so its cast and `sizeof`/`_Alignof` parsers now consume valid post-star qualifiers before checking for a second `*`; that second star produces the existing targeted `pointer-to-pointer ... not supported` boundary diagnostic rather than a generic closing-delimiter error. The parser change covers scalar plus named/anonymous aggregate type-name routes.
- 2026-07-11: No external documentation was needed for unsupported pointer-array cast diagnostics. Root cause: after `parse_cast()` consumed a supported pointer star in scalar, named/anonymous aggregate, or pointer-typedef type names, it immediately required `)`, so `(int *[2])0` reported a generic closing-delimiter error. Cust already rejects pointer arrays in declaration, typedef, `sizeof`, and `_Alignof` type routes; cast routing now checks an immediate `[` after the pointer type and reports `pointer array casts are not supported` at that token. These array cast forms are intentionally unsupported by Cust's safe one-level-pointer subset.
- 2026-07-11: No external documentation was needed for abstract function type-name diagnostics. Root cause: after consuming a supported scalar or aggregate type in `(int(void))0`, `sizeof(int(void))`, or `_Alignof(int(void))`, parser routes delegated the function declarator suffix to generic closing-parenthesis helpers. `reject_function_type_suffix()` now detects `(` not immediately followed by `*` after supported cast/type-query type names and reports a source-located unsupported-function-type diagnostic, deliberately preserving the existing parenthesized-pointer cast/type-query diagnostics.
