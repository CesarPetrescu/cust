## Findings

- 2026-06-22: No external documentation was needed for unsupported enum forward-declaration diagnostics. Implementation decision: Cust still does not support incomplete enum declarations such as `enum Color;`, but `parse_decl_type` now detects a semicolon immediately after the enum tag and reports `forward enum declarations are not supported` at the semicolon before the generic undefined-enum-type path runs.

- 2026-06-22: No external documentation was needed for the mixed declaration-context conformance fixture. Implementation decision: add a warning-free native C compiler-oracle fixture that deliberately combines already-supported surfaces—comma-separated typedef alias declarators, inline enum object declarations, pointer-first anonymous aggregate declaration lists, const anonymous aggregate arrays and pointer views, aggregate compound literal returns, and typedef-spelled aggregate return values—so future parser/runtime changes are checked against an integrated ordinary-C program. The focused interpreter test passed immediately, so no production code was changed.

- 2026-06-22: No external documentation was needed for inline enum object declarations in storage/`for` contexts. Implementation decision: allow `enum` through declaration-context routing for `for` initializer, local `static`, `auto`, `register`, `_Thread_local`, and `_Alignas` paths, then preserve inline enum constants as a preceding runtime `EnumDecl`. Static-local wrapping must recurse through `Stmt::Many` and leave `EnumDecl` unwrapped while assigning static-local ids only to actual variable declarations; otherwise inline enum constants would either be rejected after `static` or incorrectly treated as static variables.

- 2026-06-22: No external documentation was needed for comma-separated typedef alias declarator lists. Implementation decision: parse shared typedef declaration specifiers into a base `DeclType`, then parse each alias declarator with its own `*`, post-star qualifiers, and `[N]` suffix before registering parser-only alias metadata. This preserves `typedef const int *View, Table[2];` as pointer-to-const plus const array alias, and `typedef int *Ptr, * const Slot;` as ordinary pointer alias plus const pointer-slot alias. See `references/cust-comma-separated-typedef-aliases.md`.

- 2026-06-21: No external documentation was needed for same-declaration anonymous aggregate array plus pointer declarator conformance coverage. Implementation decision: existing anonymous aggregate declaration-list parsing already preserves a single unique internal anonymous type across array and pointer declarators, so this run added interpreter and native C compiler-oracle fixtures for `struct { ... } points[3] = {...}, *slot = points + 1;` and matching anonymous union arrays/pointers rather than changing runtime code. Focused RED initially exposed an incorrect expected test total; corrected fixture arithmetic to 31 before GREEN verification.

- 2026-06-21: No external documentation was needed for qualified anonymous aggregate object declarations and same-declaration anonymous aggregate pointer declarator lists. Implementation decision: route qualifier-leading aggregate declarations such as `const struct { ... } value` through `parse_aggregate_var_decl` so anonymous aggregate definitions can still allocate a unique internal type identity, then reuse the existing declaration-list tail to keep `value`, same-type copies, and `*slot = &value` pointer declarators on the same anonymous type. Standalone anonymous aggregate parameter types remain outside the supported subset because they create unnameable distinct C types.

- 2026-06-21: No external documentation was needed for anonymous aggregate object declarations. Implementation decision: reuse the same unique internal aggregate type identity path that backs anonymous aggregate typedef definitions, but do not install a user-visible tag or typedef alias for `struct { ... } object` / `union { ... } object` declarations. The parser now handles the inline aggregate body before the ordinary aggregate declarator tail, so supported global/local variables and arrays reuse existing aggregate initializer/runtime paths and C compiler-oracle coverage.

- 2026-06-21: No external documentation was needed for unsupported aggregate bit-field diagnostics. Implementation decision: Cust still does not support C bit-field storage/layout semantics, but `:` after a parsed struct/union field declarator should be detected before the generic field semicolon helper so users get targeted `bit-field aggregate fields are not supported` diagnostics for both `struct` and `union` fields. See `references/cust-aggregate-bit-field-diagnostics.md`.

- 2026-06-21: No external documentation was needed for unsupported aggregate forward-declaration diagnostics. Implementation decision: Cust still does not support incomplete aggregate types, but top-level `struct Name;` and `union Name;` should be detected before aggregate variable-declaration parsing so users get targeted `forward struct/union declarations are not supported` diagnostics rather than misleading `undefined struct type` errors. See `references/cust-aggregate-forward-declaration-diagnostics.md`.

- 2026-06-21: No external documentation was needed for typedef-backed aggregate field declaration lists. Implementation decision: when parsing aggregate field specifiers, preserve alias-carried const metadata by checking `type_alias_is_const(...)` before `parse_decl_type(...)` consumes a typedef name. This keeps `typedef int * const ConstIntSlot; struct Cursor { ConstIntSlot fixed, backup; };` as const pointer-slot fields and preserves existing `typedef const int *ConstIntView;` pointee-const metadata. See `references/cust-aggregate-field-typedef-const-metadata.md`.

- 2026-06-21: No external documentation was needed for embedded aggregate-array element copy assignment. - 2026-06-21: No external documentation was needed for direct aggregate-array element copy assignment. Detailed recovery notes live in `references/cust-aggregate-array-element-copy-assignment.md`.

Runtime pointer values remain interpreter-owned and type-checked at assignment/argument boundaries; explicit casts do not discard const-pointee safety because cast metadata preserves source constness as well as target `const`.
