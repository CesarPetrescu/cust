- [x] Enum constant declarations: `enum Tag { A = 1, B, C = -1 };` at top level and inside blocks, optional tags, implicit incrementing values, trailing commas, scoped/shadowable integer constants, read-only assignment diagnostics for enum constants, malformed-value diagnostics, and C compiler-oracle coverage
- [x] Preprocessor-free struct first milestone: top-level `struct Name { int x; char y; };` declarations, top-level/local zero-initialized `struct Name value;` variables, scalar member reads/writes with `.`, targeted unknown-field diagnostics, deterministic Cust `sizeof` for structs/fields, design notes in `docs/plans/struct-model.md`, and C compiler-oracle coverage
- [x] Struct copy assignment and field lvalue expressions: same-type `b = a;` value-copy semantics with mismatched-type diagnostics, field assignment expressions (`return p.x = 3;`), field compound assignments (`p.x += 1` and related supported operators), prefix/postfix field increment/decrement, interpreter fixtures, invalid diagnostics, and C compiler-oracle coverage
- [x] Struct by-value function parameters: parser accepts `struct Point p` parameters in definitions/prototypes after prior struct declarations, calls copy same-type struct arguments into callee scope, callee field writes do not mutate caller structs, mismatched/non-struct arguments report targeted diagnostics, and interpreter plus C compiler-oracle fixtures cover the supported subset
- [x] Struct return types: parser accepts `struct Point make(...)` definitions/prototypes after a prior struct declaration, returns clone same-type struct values by value so local return variables remain valid after callee exit, struct-returning calls can be assigned to same-type struct variables, mismatched/empty struct returns report targeted diagnostics, `sizeof(struct_return_call())` uses deterministic Cust struct sizes, and interpreter plus C compiler-oracle fixtures cover the supported subset
- [x] Struct pointers and `->`: parser accepts one-level `struct Point *p` declarations and parameters/prototypes after prior struct declarations, `&point` creates interpreter-owned struct pointers, `p->x` and `(*p).x` read/write scalar fields, field assignment/compound/increment lvalues work through pointers, null/out-of-scope diagnostics are targeted, and interpreter plus C compiler-oracle fixtures cover the supported subset
- [x] Top-level and block-scoped `typedef` aliases: `typedef int Count;`, `typedef char Byte;`, `typedef struct Point Point;`, named enum aliases such as `typedef enum Status Status;` after a prior named enum declaration, and one-level pointer aliases such as `typedef int *IntPtr;`, `typedef char *CharPtr;`, and `typedef struct Point *PointPtr;` after a prior struct declaration resolve at parse time without runtime storage changes; aliases work in globals, locals, arrays, one-level pointer declarations, function prototypes/definitions/parameters/returns, scalar struct fields, pointer-alias declarations/parameters, enum-alias declarations/parameters/returns, and `sizeof(alias)` / `sizeof(pointer_alias)`, with block-local alias and enum-tag shadowing/scope expiry, exact malformed typedef diagnostics including pointer-to-pointer alias rejection, unknown enum tag rejection, and block enum tag expiry, and C compiler-oracle coverage
- [x] First-pass `const` qualifiers for leading `const int` / `const char` scalar declarations, zero-initialized arrays, and scalar function parameters; const variables remain readable in normal expressions but direct assignment, compound assignment, increment/decrement, parameter mutation, and writes through scalar pointers are rejected, while const arrays use read-only array storage and reject indexed/pointer writes with interpreter plus C compiler-oracle coverage
- [x] `sizeof(const type)` parsing for supported scalar, typedef-alias, and pointer type forms; acceptance: `sizeof(const int) == sizeof(int)`, `sizeof(const char) == sizeof(char)`, aliases such as `Count`/`Pair` preserve their underlying size, pointer spellings such as `sizeof(const int *)` and pointer typedef aliases report deterministic pointer size, `sizeof(const void)` preserves the targeted void diagnostic, and native compiler-oracle coverage avoids ABI-sensitive exact sizes
- [x] Const-qualified pointer semantics beyond size/type contexts; acceptance: `const int *p` / `const char *p` / `const struct Point *p` create read-only pointer views that reject writes through `*p`, `p[i]`, `p->field`, `(*p).field`, compound assignment, and increment/decrement while allowing pointer reassignment; `int * const p` / `char * const p` / `struct Point * const p` create non-reassignable pointer slots that may still mutate mutable targets; `const int * const p` combines both; pointer conversions preserve pointee constness by allowing mutable-to-const pointer flow while rejecting const-to-mutable declarations/assignments/arguments; parser/interpreter metadata is documented in `docs/plans/const-pointer-model.md`; valid/invalid interpreter fixtures and C compiler-oracle coverage verify the supported subset
- [x] Const-qualified struct variables and by-value parameters for the scalar-field struct subset; acceptance: `const struct Point p;` / typedef-spelled `const Point p;` create read-only zero-initialized struct bindings, `int f(const struct Point p)` receives a by-value read-only parameter copy, direct field/copy assignment to const struct bindings reports `cannot assign to const variable '<name>'`, writes through const struct pointers or pointers to const struct targets report `cannot assign through pointer to const`, struct-pointer const-discard conversions are rejected, and valid/invalid/compiler-oracle fixtures cover the supported subset
- [x] Const-qualified scalar struct fields; acceptance: `struct Config { const int magic; const char marker; int value; };` records field-level read-only metadata, reads work through direct variables and struct pointers, mutable sibling fields remain writable, field assignment/compound/increment and pointer writes to const fields report `cannot assign to const struct field '<field>'`, whole-struct copy assignment into struct types with const fields reports `cannot assign to struct '<Type>' with const fields`, and valid/invalid/compiler-oracle fixtures cover the supported subset

- [x] Static storage-class parsing for supported top-level declarations; acceptance: lexer recognizes `static`, parser accepts it before supported global scalar/array/pointer/struct declarations plus function prototypes/definitions, and interpreter treats it as single-file linkage metadata with the same runtime behavior as ordinary globals/functions, with interpreter plus compiler-oracle fixtures covering the supported subset
- [x] Persistent local `static` storage for supported local declarations; acceptance: block/function-scope `static` scalar, pointer, array, and struct locals initialize once, persist across calls, keep lexical/block visibility, can be addressed safely after the declaring function returns through Cust's interpreter-owned pointer model, preserve const/read-only enforcement where applicable, and interpreter plus compiler-oracle fixtures cover the supported subset
- [x] Empty `void` parameter lists in function definitions and prototypes; acceptance: `int main(void)`, `int helper(void);`, and `void side_effect(void)` parse as zero-argument signatures, compatible declarations/definitions compare as empty parameter lists, malformed named/extra `void` parameters report a targeted diagnostic, and interpreter plus compiler-oracle fixtures cover the supported subset
- [x] Scalar brace initializers for one-dimensional arrays; acceptance: `int values[3] = {1, 2};`, `char bytes[4] = {1, 0, 2};`, `static int seen[3] = {4, 5};`, and `const int table[2] = {8, 9};` initialize declared elements left-to-right, zero-fill omitted trailing elements, preserve read-only const arrays and persistent static storage, accept trailing commas, reject excess initializers with `too many initializers for array '<name>'`, and include interpreter plus C compiler-oracle coverage
- [x] Scalar brace initializers for supported structs; acceptance: `struct Point p = {1, 2};`, `const struct Config c = {7, 8};`, and `static struct Point saved = {3, 4};` initialize scalar fields in declaration order, evaluate initializer expressions, zero-fill omitted trailing fields, preserve const-field/const-struct read-only enforcement after initialization, reject excess entries with `too many initializers for struct '<Type>'`, and include interpreter plus C compiler-oracle coverage

- [x] Nested struct fields for prior named struct types; acceptance: `struct Rect { struct Point origin; int width; };` stores nested field maps, `rect.origin.x` reads/writes scalar nested fields, nested fields can be copied and passed by value, and deterministic recursive `sizeof(rect.origin)` uses Cust's no-padding size model. Regression coverage includes `tests/fixtures/valid/nested_struct_fields.c`, invalid `nested_struct_unknown_field.c`, and compiler-oracle fixture `tests/fixtures/compat/valid/nested_struct_fields.c`.
- [x] Nested struct brace initializers; acceptance: `struct Rect r = {{1, 2}, 3};` recursively initializes prior named nested struct fields, zero-fills omitted nested fields, supports global/local/static/const struct declaration contexts, rejects excess nested entries with `too many initializers for struct '<Nested>'`, and includes interpreter plus C compiler-oracle fixtures.
- [x] One-dimensional scalar array fields inside structs; acceptance: `struct Packet { int values[3]; char tag[2]; };` stores array field storage with deterministic Cust element sizes, `packet.values[i]` reads/writes scalar elements, element assignment/compound/increment lvalues work, brace initializers recursively initialize array fields with omitted-element zero-fill and excess-entry diagnostics, same-type struct copy and by-value parameters deep-copy array fields, and interpreter plus C compiler-oracle fixtures cover the supported subset.

- [x] Pointer fields inside structs as the next scoped aggregate/pointer milestone; acceptance: `struct Node { int *external; struct Node *next; };` supports one-level scalar/struct pointer fields with interpreter-owned pointer metadata, self-referential struct pointer fields, pointer-field initializers/reassignment, `node.next->value` and `*node.external` use, const-pointee conversion checks for `const T *field`, targeted pointer-to-pointer/pointer-array field rejection, updated struct/pointer model docs, and valid/invalid/compiler-oracle fixtures without native ABI layout assumptions.
- [x] Arrays of supported struct types; acceptance: `struct Point points[2];` stores zero-initialized struct elements with deterministic Cust layout, supports element field access `points[i].x`, field assignment/compound/increment lvalues, brace initializers for arrays of structs, copy/by-value semantics with deep-cloned struct elements and embedded arrays while pointer fields copy pointer values, targeted bounds/initializer diagnostics, `sizeof(points)` / `sizeof(points[i])` in Cust's deterministic model, and compiler-oracle coverage that avoids native struct `sizeof` assumptions.
- [x] Address-of struct-array elements and struct fields; acceptance: `struct Point *p = &points[0];`, `&point.x`, nested field paths such as `&packet.anchor.y`, and `&points[i].x` produce interpreter-owned pointers that alias the original storage, preserve null/out-of-scope/const diagnostics, and include valid/invalid/compiler-oracle fixtures without enabling pointer-to-pointer or unsafe host addresses.
- [x] Designated initializers for supported one-dimensional scalar arrays and supported structs; acceptance: array entries such as `{[2] = 5, [0] = 1}` initialize specific elements with omitted zero-fill and mixed positional continuation, struct entries such as `{.y = 2, .x = 1}` initialize fields out of order, nested aggregate brace lists support their own designators, path designators such as `.inner.x = 1` and `.values[1] = 2` initialize nested struct fields and scalar array-field elements in source order while merging sibling nested fields, invalid array/path designator bounds and unknown struct fields report targeted diagnostics, design notes live in `docs/plans/designated-initializers.md`, and interpreter plus C compiler-oracle fixtures cover the supported subset.
- [x] Scoped `union` model as a substantial aggregate follow-up; acceptance: `docs/plans/union-model.md` documents deterministic no-ABI-padding storage semantics, named union declarations with scalar fields are supported, union variables can be zero-initialized or initialized with one first-field brace initializer, scalar field reads/writes share one logical scalar value, `sizeof(union_variable)` reports the maximum field size, excess initializers report `too many initializers for union '<Name>'`, and valid/invalid/compiler-oracle fixtures cover the supported subset.
