# Compound Literal Model

Cust supports a deliberately scoped C99-style compound literal subset:

- Supported aggregate syntax: `(struct Name){...}`, `(union Name){...}`, and aggregate typedef aliases such as `(Point){.x = 1}`.
- Supported scalar syntax: `(int){expr}`, `(char){expr}`, and scalar typedef aliases such as `(Count){expr}`. Scalar compound literals accept exactly one scalar initializer expression plus an optional trailing comma and behave like rvalue scalar expressions in Cust's existing integer storage model.
- Supported scalar-array syntax: `(int[]){...}`, `(char[]){...}`, and fixed-size forms such as `(int[3]){...}`. Array compound literals create mutable interpreter-owned array storage and evaluate as pointer expressions to element zero, so they can initialize pointer variables, decay into pointer/array parameters, and be indexed through the existing pointer-indexing model.
- Supported aggregate-array syntax: `(struct Point[]){{1, 2}, {.x = 3}}`, `(union Number[]){{7}, [2] = {.value = 9}}`, and fixed-size forms such as `(struct Point[3]){[2] = {.x = 5}}`. Aggregate-array compound literals create mutable interpreter-owned struct/union array storage in the current scope and evaluate as pointers to element zero, so existing aggregate-array pointer indexing, mutation, pointer-parameter binding, and type/const conversion checks apply.
- Aggregate brace bodies reuse the existing aggregate initializer parser, including positional entries, field designators, nested aggregate initializers, scalar array-field initializers, and union first/active-field behavior.
- Scalar brace bodies reuse the existing braced scalar initializer parser, preserving side effects exactly once in evaluated contexts and remaining non-evaluating under `sizeof`.
- Array compound literals reuse scalar array initializer rules: positional entries initialize left-to-right, omitted entries zero-fill, designators such as `[2] = 7` are supported, unsized array literals infer length from the highest initialized positional/designated element, and fixed-size literals reject excess positional or out-of-bounds designated entries with targeted diagnostics. Aggregate-array compound literals mirror those rules with aggregate element brace initializers and aggregate-array designators.
- Compound literals are rvalue expressions in Cust's interpreter model. Aggregates can initialize aggregate variables, feed same-type aggregate assignment expressions, be passed to by-value aggregate parameters, be returned from aggregate-returning functions, appear in aggregate `?:` / comma expressions, and expose scalar/nested struct fields through `.` field access. Scalars can appear in arithmetic/comparison/conditional/call-argument contexts and `sizeof((char){...})` reports Cust's deterministic char size without evaluating the initializer. Array and aggregate-array compound literals are pointer-valued rvalues; `sizeof((int[]){1, 2})` and `sizeof((struct Point[]){{1, 2}})` report Cust's pointer size.
- Cust does not model taking the address of scalar/aggregate compound literal objects or mutating scalar/aggregate compound literals as lvalues yet. Array and aggregate-array compound literals have interpreter-owned storage retained by their pointer values for the enclosing scope; future work should revisit exact C automatic-storage lifetime only if Cust introduces pointer escape diagnostics for temporary objects.
- Native C compilers remain test oracles only. Compatibility fixtures compare exit codes for supported expression behavior without relying on native ABI padding or object addresses.
